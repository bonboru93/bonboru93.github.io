TCP RACK 解析

2019-08-11

  • 基本思想

    1. 什么是乱序度

      乱序就是按123的顺序发包，收到的ACK却不是按123的顺序到来，也就是存在后发先到或者说先发后到的情况。如果现在已知后发的包先到了，那先发的包哪里去了呢？

      有两种情况，一是丢了，二是因为包在传输过程中被重新排序了，要再过一会才到。这两种情况的处理逻辑是相反的，

      a.如果包已经丢了，那么就要赶快重传。

      b.如果只是被重排了，那么再等一等他就会到，不需要重传。

      所以在出现先发后到的情况时要再选择新的表征来区分这两种情况，以便下一步采取不同的处理方法。这时候【乱序度】的概念就产生了，我们认为网络对数据包的重排是有限度的，当我们探测到乱序时，如果在限度内，认为是重排引起的；如果超过了限度，那么就认为确实是丢包了。乱序度的引入既能帮我们进行快速重传，又能通过动态变化的限度值来过滤网络原生的重排行为产生的噪声。

    2. 从包序到时序

      乱序度首先应用在包序上，协议栈根据收到的SACK信息计算乱序度，再比较窗口内的空洞情况和乱序度来决定重传多少个空洞。

      RACK则把乱序重传的概念从包序扩展到了时序。根据包序决定重传时，只能从第一个空洞开始往后重传，如果重传包又丢了，那从包序的角度上无法探知。如果从时间的维度上衡量发包的乱序，那么就能够很好的区分一个包的多次重传，一次重传后没有回音还可以再进行重传。这对包序重传的方法是一个补充。

    3. 时序乱序度怎么算

      将最近一个收到的ACK包称为RACK（recent ack），遍历发送队列（for packet in rx_packet_list），计算

       乱序度 = Δ1 + Δ2

 Δ1 = RACK.send_time - packet.send_time

 Δ2 = now - RACK.recv_time

      对比乱序度和预设限度值，决定是否重传这个packet。

      如何理解：乱序也就是先发后到、后发先到，计算两个包的乱序程度，就是【先发了多少 → Δ1】和【目前至少已经后到了多少 → Δ2】的加总。
      https://tools.ietf.org/html/draft-ietf-tcpm-rack-01 中给出的公式：

      (RACK.xmit_ts - Packet.xmit_ts) + (now - RACK.ack_ts) > RACK.reo_wnd

      也是上述含义。

    2. 4.14内核的改进

      4.4内核这样写的话等于是漏掉了乱序度概念中的后半部分，如果先后两个包形成一个突发，也就是Δ1非常小，那么在用后发包为RACK检查先发包时它就会被放过。如果此时协议栈受窗口大小或上层应用限制，得不到更多的ACK了，那么先发包就只能等待RTO了。

      4.14内核在判断时添加了一个reo定时器，也就是如果判断一个包相对RACK的乱序度还不足以立即重传它，那么可以将不足的部分视为余量，由于RACK是时间维度上的乱序，那么余量就会发生在将来。安装一个余量大小的定时器，就可以在将来再对这个包进行检查，如果检查时这个包还在发送队列中，那么就可以重传它了。

      这样不仅补完了RACK的概念，也很好的解决了当没有更多的ACK到来时如何对已经检查过但当时还不满足重传条件的包再次进行检查。这实际上也起到了代替er、thin stream等快速重传的判断逻辑的作用，因为这些逻辑应对的也是窗口受限，得不到新ACK来触发对旧包的检查时的情况。所以在上了RACK之后，time_to_recover()中的逻辑仅剩下一个包乱序的经典条件，其他全部去掉了，非常优雅。
